新建会话 · 标题延迟诊断

从点击 "+" 到左侧列表出现新条目，延迟链路的完整拆解与终极修复方案。

延迟链路 · The Latency Chain
1. 用户点击 "+" 按钮 0ms
ConversationDialog.tsx:78 → handleNewSession() 调用 create()
2. IPC: conversation_create ~5ms
Rust 端只做内存插入，已做延迟加载优化（system_prompt 推迟到首次 send）。
conversation.rs:448 — 这一步本身很快。
3. React 状态传播 ~16ms + 重渲染
setSessionId(id) 触发 useConversation 重渲染 → ConversationDialog 重渲染 → activeSessionId prop 传给 SessionList。
但 SessionList 的 sessions 数组还是旧数据，新会话不在其中。
4. 等待 3 秒轮询 ← 主要瓶颈 0~3000ms
SessionList.tsx:31-33 — setInterval(refresh, 3000)
新会话必须等到下一次轮询周期才会出现在列表中。最坏情况等待整整 3 秒。
5. conversation_list 全量磁盘 I/O ~50-200ms
conversation.rs:756 — load_session_summaries() 每次轮询都读取并反序列化 .conversations/ 下的所有 JSON 文件。
会话越多越慢。20 个会话 ≈ 读 20 个文件 + 20 次 serde 反序列化。
6. 过滤 + 渲染 ~5ms
message_count > 0 || id === activeSessionId 过滤 → 时间分组 → 渲染列表项。
这一步本身不慢。

总延迟 = 步骤 4 的轮询等待（0~3s）+ 步骤 5 的磁盘 I/O（50~200ms）≈ 最坏 3.2 秒

四个叠加的根因
根因 #1 · 致命

轮询盲区

SessionList 靠 3 秒定时器刷新，新建会话后没有任何机制触发即时刷新。用户必须等到下一个轮询周期。

SessionList.tsx:31

根因 #2 · 严重

全量磁盘扫描

conversation_list 每次调用都 read_dir + read_to_string + serde 反序列化所有会话文件。这是 O(n) 磁盘 I/O，n = 历史会话总数。

conversation.rs:161-207

根因 #3 · 中等

无乐观更新

前端 create() 拿到 session ID 后，只更新了 useConversation 内部状态，没有通知 SessionList 立即插入新条目。两个组件的状态是割裂的。

根因 #4 · 轻微

conversation-stream 事件未利用

SessionList 已经监听了 conversation-stream 的 title 和 done 事件，但没有监听 "session created" 事件（因为 Rust 端没有发射这个事件）。

终极修复方案
推荐方案
乐观插入 + 事件驱动刷新
三管齐下，彻底消除延迟：
修复 A · 即时可见

SessionList 乐观插入

  • handleNewSession 调用 create() 后，立即将一个占位 SessionSummary 插入 SessionList 的 sessions 状态
  • 方式：让 create() 返回 session ID 后，通过回调或 Rust 事件通知 SessionList
  • 最简实现：Rust conversation_create 完成后 emit 一个 session-created 事件，SessionList 监听后立即 setSessions(prev => [newSummary, ...prev])
  • 效果：从 3 秒延迟 → 0ms 延迟，新会话瞬间出现
修复 B · 降低 I/O

内存优先的 conversation_list

  • Rust 端维护一个内存中的 Vec<SessionSummary>，启动时从磁盘加载一次
  • 后续 create/delete/rename/done 时只更新内存
  • conversation_list 直接返回内存数据，零磁盘 I/O
  • 效果：每次轮询从 50-200ms → <1ms
修复 C · 降低频率

事件驱动替代轮询

  • 移除 3 秒 setInterval
  • 改为监听 Rust 事件：session-created、session-deleted、conversation-stream(title/done)
  • 仅在事件触发时刷新，或保留一个 30 秒的兜底轮询
  • 效果：CPU 和 IPC 开销大幅降低
实施顺序
Step 1 · Rust: emit session-created 事件
在 conversation_create 末尾加一行 app.emit("session-list-changed", ...)，payload 包含新会话的 SessionSummary。同样在 delete/rename 时也 emit。
Step 2 · SessionList: 监听事件 + 乐观更新
监听 session-list-changed 事件，收到后立即调用 refresh()。这一步就能把延迟从 3 秒降到 ~50ms。
Step 3 · Rust: 内存缓存 summaries（可选优化）
将 load_session_summaries 改为启动时加载一次 + 内存维护。这是锦上添花，Step 1+2 已经解决了体感问题。
Step 4 · 移除轮询（可选优化）
当事件驱动稳定后，将 3 秒轮询改为 30 秒兜底，或完全移除。

Step 1 + Step 2 是最小可行修复，预计改动 ~15 行代码，即可将体感延迟从 3 秒降到接近 0。
