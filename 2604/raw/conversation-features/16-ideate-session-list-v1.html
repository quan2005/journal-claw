竞态条件清单 4 个问题

1列表出现延迟 Rust

conversation_list 跳过 msgs.is_empty() && cancel.is_none() 的会话。新会话在 lazy prompt 构建完成前满足此条件，被过滤掉。前端 placeholder 方案是补丁，不是根治。

2列表闪烁消失 TS

前端 knownActiveRef 只在 Rust 确认后清除。但 Rust 确认的时机不确定 — 可能在 placeholder 渲染后的下一次 poll 才到，中间有一帧 placeholder 被清除但 Rust 还没返回，导致闪烁。

3分组错误 TS

Rust 返回 is_streaming: false（因为 cancel 还没设置），会话被归到"已完成"。前端用 !s.title 判断是否 streaming 是 hack，不可靠。

4数据源冲突 TS

SessionList 的数据来自两个源：Rust 轮询 + 前端合成 placeholder。两者可能冲突（重复、状态不一致），且 placeholder 的 created_at 是前端时间，与 Rust 时间可能不同步。

根因 架构

所有问题的根因是同一个：SessionList 的唯一数据源是 Rust 轮询，但 Rust 在 lazy prompt 构建完成前不知道会话已经"活跃"了。

当前的补丁方案（placeholder + knownActiveRef + debounced stream listener）试图在前端弥补这个信息差，但引入了更多复杂度和边界情况。

为什么 3s 轮询就够了？

  • 新建会话后，activeSessionId 立即更新 → 前端过滤条件 s.id === activeSessionId 保证它可见
  • 不需要等 Rust 返回 — 即使 Rust 还没返回新数据，activeSessionId 匹配就够了
  • 3s 后 Rust 返回真实数据，前端无缝替换，无闪烁
  • stream listener 的 300ms debounce 刷新只是为了让列表"快一点"出现，但有了 activeSessionId 过滤后不再需要
⚡

额外优化：轮询频率可以降低

当前 3s 轮询主要是为了检测 streaming 状态变化。但 useConversation 已经通过 streamingSessionsRef 跟踪了这个状态。可以考虑：

  • 默认 5s 轮询（降低 IPC 开销）
  • 对话框打开时 3s，关闭时停止轮询
