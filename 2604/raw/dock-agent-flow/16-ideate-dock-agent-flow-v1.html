两条主路径

路径 A — 纯文本提交

  1. 用户点击 Dock 区域或 ⌘V 粘贴短文本（≤300字符）
  2. CommandDock 打开输入态，文本填入 textarea
  3. 用户按 Enter → handleSubmit()
  4. 调用 App.handlePasteSubmit(text)
  5. 生成虚拟路径 text://{timestamp}，加入处理队列
  6. 调用 conversationCreate('agent') 创建会话
  7. 调用 conversationSend(sid, text) 发送原文
  8. Rust 端 LLM 引擎处理，通过 conversation-stream 事件回流

路径 B — 文件提交

  1. 用户拖入文件 / ⌘V 粘贴文件路径
  2. CommandDock 展示 FileCard 预览
  3. 用户可选填备注，按 Enter → handleSubmit()
  4. 调用 App.handleFilesSubmit(paths, note?)
  5. 音频文件 → importAudioFile + prepareAudioForAi（走音频管线）
  6. 非音频文件 → importFile 复制到 raw/
  7. 非音频路径 → conversationCreate('agent', undefined, paths)
  8. 调用 conversationSend(sid, prompt)，prompt 含备注

关键分支与边界情况
场景触发条件处理方式
长文本粘贴 ⌘V 粘贴 > 300 字符 调用 importTextTemp(text) 写入临时 .txt 文件，转为文件路径走文件提交路径
音频文件 文件扩展名为 m4a/wav/mp3 等 走独立音频管线：importAudioFile → prepareAudioForAi，不走 conversation 路径
混合提交 同时有音频 + 非音频文件 音频走音频管线，非音频文件合并创建一个 agent 会话
Streaming 中追加消息 会话正在 streaming 时用户再次发送 useConversation.send() 检测 isStreamingRef，走 conversationInject 而非 conversationSend
斜杠命令 ConversationInput 中输入 /summarize 等 匹配 SLASH_COMMANDS，选中后直接 onSend(cmd.promptTemplate)
会话切换 SessionList 点击其他会话 useConversation.load(id)：优先读 cacheRef（保留 streaming 状态），否则从 Rust 加载持久化消息
队列关联 文件/文本提交后 setQueueSessionId(path, sid) 将队列项与会话 ID 绑定，ProcessingQueue 可跳转到对应会话
⌘K 快捷键 全局键盘事件 切换 ConversationDialog 可见性，自动携带当前选中条目的 contextFiles
两个消息入口的区别
CommandDock（底部 Dock）是「素材投递」入口 — 用户投入原始素材（文本/文件），系统自动创建 agent 会话并处理，结果进入处理队列。用户不直接看到对话过程。

ConversationDialog（⌘K 对话框）是「交互对话」入口 — 用户直接与 Agent 对话，可以看到 streaming 过程、thinking、tool calls。支持斜杠命令、文件附件、会话历史切换。

两者共享同一套 Rust 后端（conversation.rs + llm/），但 Dock 提交后会话是「后台运行」的，而 Dialog 是「前台交互」的。Dock 创建的会话 ID 通过 setQueueSessionId 绑定到队列项，用户可以从 ProcessingQueue 点击跳转到 ConversationDialog 查看详情。
