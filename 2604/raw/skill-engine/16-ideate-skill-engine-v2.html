Skill & @ 引擎重构 v2 · 确认方案

设计哲学：把智能交给 LLM，系统只提供工具。不做路径解析、不做内容预注入、不做音频转写内联。LLM 通过 enable_skill 和 bash 自主获取一切。

Part 1 · 五条设计决策

确认的方向

统一处理逻辑

删除 resolve_skill_from_prompt()，素材处理和会话共用同一套 LLM 调用路径。不再有「素材路径特殊处理 skill」的分支。

新增 enable_skill 工具

LLM 通过调用 enable_skill 工具来读取 skill。工具返回 SKILL.md 的完整路径 + 主体内容。LLM 自主决定何时、是否激活 skill。

不做任何解析

skill 和 @ 路径都不在系统侧解析。LLM 通过 bash 自主读取文件内容。系统只在 system prompt 中列出可用 skill 清单。

前端 UI 适配

输入框支持 / 触发 skill 选择菜单、@ 触发路径选择菜单。选中后插入文本标记，但不做后端解析——纯粹是输入辅助。

tool_loop 扩展到 60 轮

从 30 轮提升到 60 轮，给 LLM 更多空间完成复杂的多步骤任务（读文件 + 分析 + 写入）。

Part 2 · 工具规格

enable_skill 工具定义

enable_skill

name: string (required)
Skill 名称，对应 skills 目录下的子目录名
返回值
JSON 对象：{ path: string, content: string }
// LLM 调用 { "name": "enable_skill", "input": { "name": "ideate" } } // 系统返回 { "path": "/Users/yan/Documents/journal/.claude/skills/ideate/SKILL.md", "content": "# 灵感探讨 (Ideate Pro Max)\n\n通过结构化..." } // 如果 skill 不存在 { "error": "skill 'foo' not found. Available: ideate, lint, visual-design-book" }

LLM 在 system prompt 中看到可用 skill 列表（名称 + 描述），当用户输入包含 /skill-name 或 LLM 判断需要某个 skill 时，主动调用 enable_skill。这比硬注入更灵活——LLM 可以在对话中途切换 skill，或同时启用多个。

Part 4 · 前端 UI 适配

/ 和 @ 的输入辅助

纯前端行为，选中后只是在输入框插入文本。不做后端解析。

Skill 选择 — 输入 / 触发
/idea
⚡ ideate 灵感探讨与设计咨询
🔍 lint 日志库自动 lint
📖 visual-design-book 可视化设计手册
@ 路径选择 — 输入 @ 触发
/ideate @2604/raw/
📄 2604/raw/meeting-notes.txt
🎙 2604/raw/recording-0416.m4a
📁 2604/raw/skill-engine/

选中 skill 后，/ideate 作为纯文本留在输入框。选中路径后，@2604/raw/meeting-notes.txt 作为纯文本插入。发送时整条消息原样传给后端，LLM 自行解读。

下一步确认

架构方向已经清晰。在开始实现之前，有一个 UX 细节想确认：

@ 路径选择的浏览深度？

A. 扁平列表 — 输入 @ 后列出当前月份 raw/ 下的所有文件（一级），不支持目录钻入。简单直接。

B. 可钻入目录 — 支持逐级浏览 workspace 目录结构（@ → 月份列表 → raw/ → 文件）。更灵活，但交互复杂度高。

C. 模糊搜索 — 输入 @mee 后模糊匹配 workspace 内所有文件名，不按目录层级。最快捷，但需要后端支持搜索。
