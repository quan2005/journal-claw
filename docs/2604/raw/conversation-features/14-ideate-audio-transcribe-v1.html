🔒

转写完成前禁止发送

发送按钮在转写期间置灰。这避免了「音频还没转完就发出去，LLM 只看到文件名」的问题。进度条给用户明确的等待预期。

👁️

转写预览 — 可折叠

转写完成后在 chip 下方展示前几行文本预览，让用户确认转写质量。点击可展开全文。如果转写有误，用户可以移除重试或手动编辑。

♻️

复用现有 ASR 管道

调用 transcription.rs 的现有函数，优先检查是否已有 .transcript.json sidecar（同一音频可能在日志处理时已转写过）。有则直接读取，无则触发新转写。零重复工作。

📨

注入方式 — 结构化文本

发送给 LLM 时，转写文本以结构化格式注入 user message：
[音频转写: 会议录音.m4a]\nSpeaker A (0:00): ...\nSpeaker B (1:23): ...\n[/音频转写]\n\n用户提问: 帮我总结...
这样 LLM 能区分转写内容和用户指令。

📄

扩展 — 其他文件类型同理

同样的模式可以扩展到 PDF、DOCX：附件先经过内容提取（pdftotext / pandoc），再注入对话。统一的「附件 → 文本 → LLM」管道。
