Dock → Agent 架构改进方案

问题一：串行队列持久化 · 问题二：Dock / Dialog 路径统一

问题一 · 串行队列持久化

当前问题 现状

  • workQueueRef 是纯 useRef，页面刷新即丢失
  • queueItems 是 useState，同样不持久
  • 用户投入 5 个文件，第 1 个正在处理时刷新页面 → 后 4 个全部丢失
  • 正在处理的会话（Rust 端仍在运行）前端失去追踪

目标 修复

  • 页面刷新后，待处理的 WorkItem 不丢失
  • 正在处理的会话能重新关联
  • 已完成/失败的项保留短暂可见
  • 不引入额外的 Rust 端复杂度（尽量前端解决）
关键洞察：Rust 端的 ConversationSession 已经持久化到 .conversations/{id}.json。问题不在于会话丢失，而在于前端的「队列 → 会话」映射关系丢失。修复的核心是持久化这个映射。
方案对比

方案 A · localStorage

将 workQueueRef 和 queueItems 序列化到 localStorage。

  • 改动最小，纯前端
  • 刷新后从 localStorage 恢复队列
  • 需要处理「恢复后会话已完成」的情况
  • localStorage 有 5MB 限制，但队列数据很小
  • 多窗口场景可能冲突

方案 B · Rust 端队列 推荐

将工作队列下沉到 Rust，前端只做 UI 展示。

  • 新增 Rust 命令 enqueue_work / list_work_queue
  • Rust 端维护队列 + 串行执行 + 持久化到 workspace
  • 前端刷新后通过 list_work_queue 恢复
  • 天然支持多窗口（Rust state 是单例）
  • 改动较大，需要新增 Rust 模块

方案 C · 混合方案

队列逻辑留在前端，但通过 Rust 命令持久化映射关系。

  • 新增 save_work_queue / load_work_queue 命令
  • 前端 enqueueWork 时同步写入 Rust
  • 刷新后从 Rust 加载恢复
  • 串行逻辑仍在前端（processNextWork）
  • 两端都有队列概念，容易不一致

推荐：方案 B — Rust 端队列

理由：串行队列本质上是「后台任务调度」，这是 Rust 进程的职责而非 webview 的职责。 当前 processNextWork 在前端用 await new Promise 阻塞等待 conversation-stream 事件， 这意味着前端 JS 线程被占用来做调度 — 如果 webview 刷新，调度就断了。

将队列下沉到 Rust 后，前端变成纯展示层：调用 enqueue_work 投递任务，监听事件更新 UI，刷新后调用 list_work_queue 恢复状态。

// ── 新增 Rust 模块：src-tauri/src/work_queue.rs ── pub struct WorkQueue { items: Vec<WorkItem>, processing: Option<String>, // 正在处理的 item id } pub struct WorkItem { id: String, status: WorkStatus, // queued | processing | completed | failed session_id: Option<String>, // 关联的 conversation session text: Option<String>, files: Option<Vec<String>>, prompt: Option<String>, display_name: String, // UI 显示名 created_at: u64, } // Tauri 命令 fn enqueue_work(text, files, prompt, display_name) → WorkItem fn list_work_queue() → Vec<WorkItem> fn cancel_work_item(id) fn retry_work_item(id) fn dismiss_work_item(id) // 事件 "work-queue-updated" → 前端监听，刷新队列 UI
// ── 前端变化：useJournal.ts ── // 删除：workQueueRef, processingWorkRef, processNextWork, enqueueWork // 删除：queueItems useState（改为从 Rust 获取） // 新增： const [queueItems, setQueueItems] = useState<WorkItem[]>([]) // 初始化时加载 useEffect(() => { listWorkQueue().then(setQueueItems) }, []) // 监听更新事件 listen('work-queue-updated', () => { listWorkQueue().then(setQueueItems) }) // App.tsx 中： // handlePasteSubmit: enqueueWork(...) → invoke('enqueue_work', ...) // handleFilesSubmit: enqueueWork(...) → invoke('enqueue_work', ...)
问题二 · Dock / Dialog 路径统一讨论

当前架构：两条独立路径

CommandDock（素材投递）ConversationDialog（⌘K）
状态管理useJournal.enqueueWork + queueItemsuseConversation + messages
会话创建在 processNextWork 内部在 useConversation.create
事件监听一次性 listener（done/error）全局 listener（所有事件）
UI 展示ProcessingQueue（状态条）ConversationDialog（完整对话）
用户感知后台静默处理前台交互对话
↓ 统一方向 ↓

选项 X · 最小统一

保持两条路径分离，但共享会话层。

  • Dock 和 Dialog 都通过 useConversation 创建/管理会话
  • Dock 创建的会话标记为 background: true
  • ProcessingQueue 从 useConversation.sessions 过滤 background 会话
  • 删除 useJournal 中的队列逻辑，串行调度移到 Rust
  • 改动适中，风险低
  • Dock 提交的会话可以无缝在 Dialog 中打开

选项 Y · 完全统一

Dock 提交后直接打开 ConversationDialog。

  • Dock 变成纯「素材导入 + 会话创建」入口
  • 提交后自动弹出 Dialog 显示处理过程
  • 不再有「后台处理」概念，所有会话都是前台
  • ProcessingQueue 退化为「最近会话」列表
  • 用户投入 5 个文件会弹出 5 个对话？UX 问题
  • 破坏「投入即忘」的轻量体验

推荐：选项 X — 最小统一

完全统一（选项 Y）会破坏 Dock 的核心体验 —「投入素材，忘掉它，稍后看结果」。 这是日记工具的关键交互：用户不想每次投入文件都被拉进一个对话。

最小统一的核心思路：让 useConversation 成为唯一的会话状态源，useJournal 不再管理会话生命周期。 具体来说：

// ── 统一后的数据流 ── // 1. Dock 提交 CommandDock → App.handlePasteSubmit() → invoke('enqueue_work', { text, files, prompt }) // Rust 队列 → Rust 串行处理，自动 conversation_create + conversation_send → emit 'work-queue-updated' // 前端更新队列 UI → emit 'conversation-stream' // useConversation 自动捕获 // 2. Dialog 对话 ConversationInput → useConversation.send() → invoke('conversation_send', { sessionId, message }) → emit 'conversation-stream' // useConversation 处理 // 3. 从队列跳转到对话（已经可行，无需改动） ProcessingQueue click → setConversationState({ initialSessionId: item.sessionId }) → ConversationDialog.load(sessionId) → useConversation.cacheRef 已有该会话的 streaming 数据 ✓
