Skill & @ 文件引擎 — 架构探讨

梳理当前 AI 处理引擎中 skill 加载和文件引用的完整数据流，识别架构断层，探讨统一设计方案。

Part 1 · 现状

两条割裂的处理路径

当前存在两条独立的 LLM 调用路径，skill 系统只在其中一条生效，@ 文件引用则完全是文本约定。

素材处理 拖入文件 / 录音 → ai_processor → resolve_skill ✓ → build_system_prompt (含 skill) → tool_loop::run_agent → Anthropic API
会话聊天 用户输入 /skill → conversation.rs → ❌ 无 skill 解析 → build_system_prompt (无 skill) → run_conversation_turn → Anthropic API
@ 文件 @yyMM/raw/file.m4a → 纯文本，无解析 → LLM 自行 bash cat
Part 2 · 架构断层

四个需要解决的问题

会话路径不支持 Skill

conversation.rs 的 run_conversation_turn() 从不调用 resolve_skill_from_prompt()。用户在聊天中输入 /ideate 会命中前端的 5 个硬编码斜杠命令，而非后端 skill 系统。

前端斜杠命令 ≠ 后端 Skill

slashCommands.ts 维护了 5 个静态 prompt 模板（summarize/todos/translate/rewrite/tags），与 skills.rs 扫描的 SKILL.md 完全独立。两套系统，两个入口，零交集。

@ 文件是「君子协定」

ai_processor.rs 生成 @yyMM/raw/filename 纯文本，期望 LLM 自己用 bash 读取。没有前端 UI 支持，没有路径校验，没有内容预注入。LLM 可能读错路径或忽略 @。

内置 Skill 每次启动覆写

ensure_workspace_dot_claude() 用 include_str! 在每次启动时覆盖 ideate/lint/visual-design-book。用户对这些 skill 的自定义修改会丢失。

Part 4 · 设计方向

统一 Skill 与 @ 文件的三种路径

核心问题：如何让会话路径也能激活 skill、解析 @ 文件引用，同时保持两条路径的架构清晰？

方案 A

共享解析层 — 前置统一

在 conversation.rs 和 ai_processor.rs 之前，抽取一个共享的 prompt_resolver 模块，负责：

  • 解析 /skill-name → 加载 SKILL.md 内容
  • 解析 @path/to/file → 读取文件内容，注入为上下文
  • 前端斜杠命令合并为 skill（迁移 5 个硬编码命令为 SKILL.md）

改动范围：新增 prompt_resolver.rs，修改 conversation.rs + ai_processor.rs + slashCommands.ts

  • 两条路径行为一致，用户心智模型统一
  • @ 文件内容预注入，不依赖 LLM 猜路径
  • 前端只需维护一套 skill 列表（从后端动态获取）
  • 需要定义 @ 解析的边界（多大的文件？二进制？）
方案 B

会话侧独立增强

仅在 conversation.rs 中添加 skill 解析和 @ 文件处理，不抽取共享模块。两条路径各自维护解析逻辑。

  • conversation.rs 新增 resolve_skill 调用
  • @ 文件在 conversation.rs 中独立解析
  • 前端斜杠命令保持不变

改动范围：仅修改 conversation.rs

  • 改动最小，风险最低
  • 不影响已稳定的素材处理路径
  • 两处重复逻辑，未来维护成本高
  • 前端仍有两套斜杠命令系统
方案 C

前端统一入口

将 skill 解析和 @ 文件处理全部移到前端。前端负责：识别 /skill → 调用后端获取 SKILL.md 内容 → 注入到消息中发送。

  • 新增 Tauri 命令 get_skill_content(name)
  • 新增 Tauri 命令 read_file_content(path)
  • ConversationInput 解析 / 和 @ 后预处理消息

改动范围：新增 2 个 Tauri 命令，重写 ConversationInput + slashCommands

  • 前端完全掌控 UX（自动补全、预览）
  • 后端保持纯粹的 LLM 调用
  • skill 内容通过 IPC 传输，增加延迟
  • 素材处理路径仍需后端解析，又回到两套

聚焦讨论

在深入设计之前，我想先确认一个关键决策：

@ 文件引用的解析深度应该到哪一层？

A. 浅解析 — 只做路径校验 + 将文件路径转为绝对路径，仍然让 LLM 通过 bash 工具自行读取内容。好处是简单，且 LLM 可以决定读取策略（全文 / 部分 / 跳过）。

B. 深解析 — 读取文件内容并注入到消息上下文中（文本直接内联，音频先转写再内联，二进制给出元信息）。好处是 LLM 不需要浪费工具调用轮次去读文件，响应更快。

C. 混合策略 — 小文件（<50KB 文本）深解析内联，大文件 / 二进制浅解析只给路径提示。兼顾效率和 token 成本。
