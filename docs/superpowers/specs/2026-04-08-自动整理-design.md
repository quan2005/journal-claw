# 自动整理 — 设计规格

## 概述

在设置页新增「自动整理」Section，让 dream 技能（日志库自动整理）从手动 `/dream` 触发变为可配置的定时自动执行。

## 用户故事

知识工作者每天产生多条日志，日志库随时间积累会出现交叉引用缺失、决策演进未标注、Identity 档案过时等「记忆腐烂」问题。用户希望这些整理工作自动发生，不需要记得手动触发。

## 设计决策

| 决策 | 选择 | 理由 |
|---|---|---|
| 执行方式 | 应用内后台静默执行 | dream 本身安全（不删除、只追加），静默执行符合「打开即平静」 |
| UI 位置 | 独立「自动整理」Section | 与 Plugins 平级，为未来更多自动化任务预留空间 |
| 布局 | 紧凑单卡 | 所有配置在一张卡片内，信息密度合适 |
| 触发逻辑 | 定时 + 条目门槛双条件 | 定时器到点时检查新增条目数，达标才执行，避免无意义空跑 |
| 运行状态 | 状态行文字变化 | 最小侵入，不需要进度条 |

## UI 规格

### 导航栏

在现有导航中「权限」和「插件」之间插入「自动整理」项：

```
通用 → AI 引擎 → 语音 → 权限 → 自动整理 → 插件 → 关于
```

图标：使用 `lucide-react` 的 `Timer` 图标。

### 自动整理 Section — 紧凑单卡

卡片内从上到下：

1. **头部**：图标（🗂）+ 标题「日志库自动整理」+ 描述 + 开关 toggle
2. **分隔线**
3. **整理频率**：segment 选择器，三个选项 — 每天 / 每周 / 每月
4. **执行时间**：segment 选择器，三个选项 — 凌晨 3:00 / 中午 12:00 / 晚间 22:00
5. **最少新增条目**：segment 选择器，三个选项 — 10 篇 / 20 篇 / 30 篇，后跟「不足则跳过」提示
6. **分隔线**
7. **状态行 + 立即整理按钮**：
   - 左侧：上次整理时间 + 整理篇数；下次检查时间 + 当前新增篇数
   - 右侧：「立即整理」按钮

### 状态行变化

| 状态 | 显示 |
|---|---|
| 空闲 | 上次: YYYY-MM-DD HH:mm · 整理了 N 篇 / 下次检查: YYYY-MM-DD HH:mm · 当前新增 N 篇 |
| 执行中 | 正在整理中... |
| 从未执行 | 尚未执行过自动整理 |
| 上次失败 | 上次整理失败: 错误信息 |

### 开关行为

- 开启：启动定时器，按配置的频率和时间调度检查
- 关闭：停止定时器，配置保留不丢失
- 「立即整理」：无视条目门槛，直接执行 dream 技能；执行中按钮变为 disabled 状态

## 数据模型

### workspace `.setting.json` 扩展

```json
{
  "theme": "dark",
  "auto_dream": {
    "enabled": true,
    "frequency": "daily",
    "time": "03:00",
    "min_entries": 10
  }
}
```

字段说明：

| 字段 | 类型 | 可选值 | 默认值 |
|---|---|---|---|
| `enabled` | bool | — | `false` |
| `frequency` | string | `"daily"` / `"weekly"` / `"monthly"` | `"daily"` |
| `time` | string | `"03:00"` / `"12:00"` / `"22:00"` | `"03:00"` |
| `min_entries` | number | `10` / `20` / `30` | `10` |

### 运行状态来源

- 上次整理时间和条目数：读取 workspace `.claude/last-dream.json`（dream 技能自己维护）
- 当前新增条目数：当前总条目数 - `last-dream.json` 中的 `entries_at_last_run`

## 后端逻辑

### Rust 模块变更

#### `workspace_settings.rs`

扩展 `WorkspaceSettings` struct，增加 `auto_dream` 字段：

```rust
#[derive(Debug, Serialize, Deserialize)]
struct AutoDreamConfig {
    #[serde(default)]
    enabled: bool,
    #[serde(default = "default_frequency")]
    frequency: String,    // "daily" | "weekly" | "monthly"
    #[serde(default = "default_time")]
    time: String,         // "03:00" | "12:00" | "22:00"
    #[serde(default = "default_min_entries")]
    min_entries: u32,     // 10 | 20 | 30
}
```

新增 Tauri commands：
- `get_auto_dream_config` → `AutoDreamConfig`
- `set_auto_dream_config(config: AutoDreamConfig)` → `()`

#### 新模块 `auto_dream.rs`

定时调度器，职责：

1. **启动**：app 启动时读取配置，如果 `enabled` 则启动定时器
2. **调度检查**：定时器到点时：
   - 读取 `.claude/last-dream.json` 获取 `entries_at_last_run`
   - 扫描当前总条目数
   - 如果新增数 ≥ `min_entries`，执行 dream
   - 否则跳过，等下次
3. **执行 dream**：复用 `ai_processor.rs` 的 CLI 调用模式（`tokio::process::Command`，`.current_dir(workspace_path)`，注入 credentials/model/PATH），prompt 为 `"/dream"`，Claude CLI 会自动发现 workspace 下的 `.claude/skills/dream/skill.md` 并执行
4. **状态事件**：执行开始/完成/失败时 emit Tauri event `"auto-dream-status"`
5. **配置变更响应**：前端调用 `set_auto_dream_config` 时，重启或停止定时器

#### 定时器实现

使用 `tokio::time::interval` 或 `tokio::time::sleep_until`。每次 app 启动时计算下次执行时间点：

- `daily`：今天或明天的指定时间
- `weekly`：下一个周日的指定时间（周末整理一周的积累）
- `monthly`：下月 1 号的指定时间

到点后执行检查，然后计算下一个时间点继续 sleep。

#### 新增 Tauri commands

| Command | 参数 | 返回 | 说明 |
|---|---|---|---|
| `get_auto_dream_config` | — | `AutoDreamConfig` | 读取配置 |
| `set_auto_dream_config` | `AutoDreamConfig` | — | 保存配置并重启/停止定时器 |
| `get_auto_dream_status` | — | `AutoDreamStatus` | 当前状态（idle/running/last_run 信息） |
| `trigger_dream_now` | — | — | 立即执行，无视条目门槛 |

#### `AutoDreamStatus` 结构

```rust
#[derive(Debug, Serialize, Deserialize)]
struct AutoDreamStatus {
    state: String,              // "idle" | "running" | "never_run" | "error"
    last_run: Option<String>,   // ISO timestamp
    last_run_entries: Option<u32>,
    next_check: Option<String>, // ISO timestamp
    current_new_entries: u32,
    error: Option<String>,
}
```

## 边界情况

- **app 未运行时到达执行时间**：下次启动时检查是否错过，如果错过且条目达标则立即执行
- **执行中关闭 app**：Claude CLI 子进程被 kill，下次启动视为未完成，不更新 `last-dream.json`
- **workspace 路径变更**：重新读取新 workspace 的配置和 `last-dream.json`
- **Claude CLI 不可用**：执行失败，状态行显示错误，不影响定时器继续调度
- **并发**：如果用户点「立即整理」时定时任务也触发了，用 mutex 保证同一时间只有一个 dream 在跑
