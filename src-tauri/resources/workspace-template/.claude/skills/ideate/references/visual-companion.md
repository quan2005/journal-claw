# Visual Companion — Canvas 规范与模板用法

ideate 的可视化画板只解决一件事：**把一个探索中的设计问题用最精练的视觉讲清楚**。它不是成品展示墙，也不是设计稿；它是讨论的辅助工具。

## 核心美学：Quieter Canvas

每个像素都有意图，但绝不张扬。这意味着：

- **点阵背景** — 营造"设计画板"的隐喻，而不是白纸或深色仪表盘
- **排版建立层级** — 靠字阶与行高分层，不靠颜色或装饰
- **单色调 + 一个 accent** — 全局只有一个干净的强调色承担视觉焦点
- **Subtle interactions** — hover 用平滑阴影，active 轻微缩小（0.98），不要 bounce

### 必须做到

- **8pt/4pt 呼吸间距** — 组内紧凑（gap 8–16px），组间松弛（24–40px）
- **对比度 ≥ 4.5:1**（WCAG AA），含辅助文本
- **可点击元素有视觉反馈** — 只对 `transform`、`box-shadow`、`border-color` 施加过渡
- **降级处理** — 加入 `@media (prefers-reduced-motion: reduce)`
- **内联注入 `scripts/canvas.css`** — 每次画板视觉一致的唯一方式

### 禁止出现

- **金橙色系 `#C8933B`、`Noto Sans SC` 字体、1100px 宽布局** — 这些属于 visual-design-book 的视觉体系，用了就走味了
- **`transition: all`** — 过渡应明确针对 `transform` / `box-shadow` / `border-color`
- **纯黑 `#000` / 纯白 `#FFF` 大面积背景** — 用细腻的浅灰 / 深灰替代
- **紫蓝渐变、霓虹、过度背景模糊、渐变文字** — AI slop 视觉
- **Hero → 北极星 → 逐章推演 → 终章收束的叙事结构** — 那是 visual-design-book 的结构

## 何时用浏览器 vs 终端

不要因为能用浏览器就用它。只在"视觉比文字更能降低认知负荷"时切换。

**用浏览器：**

- **布局选型** — A/B 方案并排对比
- **视觉风格** — 颜色板、排版层级、组件形态探讨
- **组件微动效** — 需要展示 hover/active/focus 的具体表现
- **流程图与信息架构** — 文字过长时用 Bento Grid 形式的卡片流程图

**用终端：**

- 需求梳理、概念探讨、技术栈决策、优劣势逻辑罗列

## 画板生成流程

1. **选模板** — 从 `scripts/` 里挑最匹配讨论场景的模板文件（见下方清单）
2. **写新文件** — 按 SKILL.md 的路由规则确定输出路径，**不要复用文件名**，每次迭代写 `v2`、`v3`
3. **注入 canvas.css**：
   - 用 `Read` 工具读 `scripts/canvas.css` 的完整内容
   - 把它整段包在 `<style>` 标签里注入 HTML
4. **替换占位符** — 填充你的设计内容
5. **用 `open` 命令打开** — 在浏览器里全屏展示
6. **终端反馈** — 用一句话告诉用户你展示了什么、请他选哪个
7. **迭代** — 收到反馈后写 `v{n+1}`，不覆盖旧文件

## 模板清单

模板位于 `scripts/` 目录（历史命名，本质是资源文件）。

### `ab-test.html` — 方案对决

- **场景**：选哪个方案、概念对比、优劣势对比
- **组件**：极简选项卡片（带 Badge 和选中态）+ Bento Grid Pros/Cons
- **适用对话**：「选哪种侧边栏结构？」「选哪种存储策略？」

### `wireframe.html` — 布局线框

- **场景**：UI 布局推演、信息架构、组件结构探索
- **组件**：macOS 风格画板容器（红黄绿控制点）+ 移动端骨架框 + 各种 Skeleton 积木
- **适用对话**：「首页怎么排布？」「这个表单层级对吗？」

### `styleguide.html` — 视觉风格

- **场景**：设计令牌、配色、排版探索
- **组件**：高保真色板 + 排版阶层样本
- **适用对话**：「品牌主色选哪个？」「用哪种字体组合？」

### `flow.html` — 流程图

- **场景**：用户旅程、状态机、数据流
- **组件**：选中态卡片节点 + 平滑箭头 + 垂直步骤列表
- **适用对话**：「登录注册的最佳转化路径？」「API 请求时序？」

### `bento.html` — Bento Grid

- **场景**：复杂页面拆解、多维度信息汇总
- **组件**：网格化圆角卡片组合

## 画板质量心法

- **Fidelity Matching** — 早期需求保真度低（多用 `wireframe.html` 的色块），讨论细节时才用精细排版
- **Real Content is King** — 用真实文案代替 Lorem Ipsum，大幅提升设计真实感
- **Bento 优先** — 复杂页面不要平铺，用网格 + 圆角卡片组合的 Bento Box
- **深浅主题都检查** — 模板默认有 `prefers-color-scheme: dark` 支持，自己新增元素要测深浅模式边界
