# 画像提炼规范

## 契约

| 维度 | 声明 |
|---|---|
| **reads** | `identity/*.md`、`yyMM/*.md`（提取人物信息的源素材） |
| **writes** | `identity/*.md` |
| **depends** | 无 |

档案存在的唯一理由：**让 AI 秘书下次遇到这个人 / 产品时，能更准确地理解上下文、写出更好的日志。** 不是存档、不是回忆录。

每次动 identity/ 档案前，回到这个问题：读者（未来的你或 AI 秘书）在什么情况下会打开这个档案？他想找到什么？按这个标准过滤信息。

## 一、人物画像

### 提炼五维度

每次从素材中提取人物信息，沿五个维度归类。不要按会议日期追加段落。

| 维度 | 要回答的问题 | 性质 |
|---|---|---|
| **身份与位置** | 这个人是谁、在组织里站在哪里、和用户是什么关系 | 事实性，每次有新信息就更新 |
| **关切与议程** | 这个人当前最在意什么、在推动什么 | 事实性，每次有新信息就更新 |
| **决策模式** | 怎么做判断、什么能说服 ta、什么会触发反对 | 归纳性，从多次事件中提炼模式 |
| **能力与盲区** | 擅长什么、在哪些方面判断可能有偏差 | 归纳性，从多次事件中提炼模式 |
| **与用户的杠杆关系** | 能帮用户什么、需要用户什么、可能阻碍用户什么 | 归纳性，需要跨事件综合判断 |

### 渐进深挖规则

不是所有人都需要五维度全写。按出现次数和互动深度分级：

| 阶段 | 触发条件 | 写什么 |
|---|---|---|
| **初始化** | 首次出现 | 身份与位置 + 杠杆关系（两三句话） |
| **补充** | 第二次出现 | 追加「关切与议程」 |
| **深挖** | 出现 3 次以上，且有决策行为或立场表达 | 提炼「决策模式」和「能力与盲区」 |
| **永远停在初始化** | 纯信息传递角色（被指派做某事、只说了一句话） | 不深挖 |

**判断标准**：如果这个人下次出现在素材里，AI 秘书是否需要知道 ta 的决策风格才能写好日志？需要就深挖，不需要就停。

### 三层信息:事实 / 模式 / 边界

档案里的每条信息必须能归到下面三层之一。混层写是档案腐烂的起点。

| 层 | 是什么 | 何时可以写 | 例子 |
|---|---|---|---|
| **事实** | 直接从素材中观察到的、不需要推断的信息 | 只要出现一次就可以写 | 职位、部门、汇报关系、某次会议上的原话 |
| **模式** | 从多次事件归纳出的决策风格 / 表达习惯 / 价值倾向 | 满足下面「三重验证」才能写 | 「遇到卡点直接拍板」「对复杂度有直觉性警觉」 |
| **边界** | 明确标注哪些维度无法可靠推断、推断可能失效的情境 | 当你发现自己想补白但证据不足时 | 「私下场合的态度未观察到」「技术判断仅在 X 领域可靠」 |

#### 模式的三重验证(升级为"模式"前必须同时满足)

一条归纳想从"观察"升格为"模式"、写进档案并用于未来推断,必须同时满足:

1. **跨场景复现** — 至少 3 次**独立**事件中表现一致(同一场会议的多次发言不算独立)
2. **生成力** — 这个模式能帮你**预测**此人在未来新情境中的反应,而不只是描述过去
3. **排他性** — 这个描述指向的是**这个人的特征**,而不是在这种角色 / 这家公司任何人都会有的通用行为

不满足任何一条 → 留在"事件记录 / 待观察",不要写入决策模式维度。

**反例**(不满足排他性):
> ❌ 决策模式:注重数据驱动
> (任何产品经理都会这么说,没有刻画这个人)

**正例**(满足三重):
> ✅ 决策模式:对技术复杂度有直觉性警觉,会追问「为什么不用更简单的方案」
> (4/10 追问飞书替代、4/13 挑战架构分层、5/02 否决自建鉴权 — 三次独立场景复现,能预测下次遇到新方案时会提"最简路径",且不是同类角色通用反应)

#### 诚实边界(升级为"模式"时同步写出)

每当在"决策模式""能力与盲区""与用户的杠杆关系"维度写入一条归纳,必须同时思考这条归纳**在什么情境下会失效**。如果存在明显边界,显式写在档案的「观察边界」段落:

```markdown
## 观察边界
- 只观察到工作场景的决策风格,私下场景未知
- 技术判断仅在后端 / 架构领域有证据,对前端 / 设计领域的判断力未验证
- 所有互动均通过会议观察,未见过 1-on-1 或书面决策文档
```

不写边界 = 假装档案是全景画像。下次 AI 秘书拿这份档案去推断"此人对设计方案会如何反应"时,会做出超出证据范围的判断。

### 档案结构

模板在 `assets/templates/person.md`。创建新人物档案时读取模板、按渐进深挖规则填充。

### 关键写作原则

#### 按维度组织，事件是证据而非主体

错误示范：

```markdown
## 4/10 碰撞会补充
- 对架构复杂度有直觉性担忧
- 追问"为什么不直接用飞书"

## 4/13 权限讨论会
- 首次完整表达"盖新楼"愿景
```

正确示范：

```markdown
## 决策模式
- 遇到卡点不纠结流程，直接推动决策落地（4/10 当场拍板飞书权限；4/13 推动蔡旭从保守转务实）
- 对技术复杂度有直觉性警觉，会追问「为什么不用更简单的方案」（4/10 追问为什么不直接用飞书）

## 关切与议程
- 北极星：用 AI 重构 HR 工作范式——「盖新楼」而非改旧楼（4/13 首次完整表达）
- 当前第一优先级：快速让人用起来，而非完美体验
```

**区别**：维度是骨架，事件是肌肉。读者查档案时按维度找答案，不需要按时间线翻阅。

#### 归纳而非罗列

同一个人在三次会议中都表现出「务实推动」的风格时，不要写三段会议记录，而是写一条归纳结论 + 三个括号内的证据引用。

#### 原话引用的标准

只保留满足以下任一条件的原话：

- 暴露了此人的核心价值观或决策逻辑
- 是一个可复用的判断框架（如「不要告诉我你问了谁，告诉我答案是什么」）
- 用词精准到无法用转述替代

其余一律用归纳语言转述。

### 更新策略

每次处理涉及已有档案人物的素材：

1. 先读档案，理解当前画像
2. 从素材中提取新信息，判断属于哪个维度
3. **合并而非追加** — 把新信息融入对应维度的已有描述，而不是在底部加日期段落
4. 新信息推翻旧描述（如职位变化）→ 直接更新，不保留旧版本
5. 新信息丰富归纳结论 → 在括号中追加新的证据引用

## 二、产品画像

### 是否建档（准入规则）

产品档案的建档门槛高于人物——默认把信息留在日志条目内。仅在**同时满足**以下三条时才创建档案：

1. **复现性** — 在 2 条以上独立素材中出现（同一场会议多次提及不算）
2. **回溯价值** — 未来写日志时需要其背景（产品定位、关键决策）
3. **演化属性** — 有随时间变化的属性（版本、阶段、负责人、关键决策）

存疑则不建档。档案命名 `product-{name}.md`（如 `product-mobo.md`），frontmatter 加 `type: product` 与人物档案区分。

### 骨架与状态分离

产品档案里有两类信息，衰减速度完全不同：

| 类型 | 衰减周期 | 示例 |
|---|---|---|
| **骨架**（Architecture） | 季度级 | 产品定位、架构分层、团队归属、与其他产品的关系 |
| **状态**（State） | 周级 | 版本进度、客户商机阶段、当前阻塞项 |

骨架写在档案主体 — 是产品的「身份证」。状态集中在一个段落，标注更新日期，每次**整体刷新**而非追加。

### 提炼四维度

| 维度 | 要回答的问题 | 衰减速度 |
|---|---|---|
| **定位与边界** | 解决什么问题、不解决什么问题、在产品矩阵里的位置 | 慢 |
| **核心能力与架构** | 技术栈、能力分层、关键性能指标 | 慢 |
| **战略判断与关键决策** | 谁在什么时候做了什么定调、为什么 | 慢（标注决策时间） |
| **当前态势** | 版本进度、客户管线摘要、阻塞项、近期风险 | 快 |

### 档案结构

模板在 `assets/templates/product.md`。

### 关键写作原则

- **当前态势整体刷新，不追加** — 每次处理涉及该产品的素材时，重写整个「当前态势」段落并更新日期。旧状态已经在日志条目里了
- **客户商机不展开** — 产品档案只保留一行摘要（如「台山核电：合同审批中」）。详细进展放日志
- **战略决策标注决策人和时间** — 产品方向可能被推翻，标注来源让读者判断决策是否仍有效
- **产品间关系双向引用** — 一条信息涉及两个以上产品的关系时，两个档案都写入引用

## 三、跨档案规则

### 产品决策归产品档案

某人在会议中做了产品级决策（如侍洁拍板 MVP 范围）：

- 写入**产品档案**的「战略判断与关键决策」
- **人物档案**只保留归纳后的决策模式（如「遇到卡点直接拍板」），不展开产品细节

错误：在侍洁档案里写三段关于墨宝 MVP 范围的讨论细节
正确：墨宝档案记录 MVP 决策，侍洁档案归纳决策风格

### 事实性信息的唯一权威源

| 信息类型 | 权威源 | 其他地方怎么写 |
|---|---|---|
| 某人的职位 / 角色 | 人物档案 | 日志中首次提及时标注，不重复维护 |
| 产品的定位 / 架构 | 产品档案 | 日志中引用，不重复描述 |
| 某次会议的具体讨论 | 日志条目 | 档案中用括号引用日志路径 |
| 商机的详细进展 | 日志条目 | 产品档案只保留一行摘要 |

### 过时信息处理

**不要在档案中追加 ⚠️ 过时警告文本**。改为：

- 信息确认仍有效 → 不做任何标注
- 信息已过时 → 直接更新为最新状态，或删除已不相关的内容
- 无法确认 → 在 lint 摘要中列出供人工复核，不在档案中标注
