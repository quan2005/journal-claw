# 画像提炼规范

档案存在的唯一理由：让 AI 秘书下次遇到这个人/产品时，能更准确理解上下文。
不是存档，不是回忆录。

---

## 一、人物画像

### 五维度

| 维度 | 回答什么 | 性质 |
|---|---|---|
| 身份与位置 | 是谁、站哪里、和用户什么关系 | 事实，有新信息就更新 |
| 关切与议程 | 当前最在意什么、在推动什么 | 事实，有新信息就更新 |
| 决策模式 | 怎么判断、什么能说服、什么触发反对 | 归纳，需三重验证 |
| 能力与盲区 | 擅长什么、判断在哪里会有偏差 | 归纳，需三重验证 |
| 与用户的杠杆关系 | 能帮什么、需要什么、可能阻碍什么 | 归纳，跨事件综合判断 |

### 渐进深挖

| 阶段 | 触发 | 写什么 |
|---|---|---|
| 初始化 | 首次出现 | 身份与位置 + 杠杆关系（2–3 句） |
| 补充 | 第二次出现 | 追加「关切与议程」 |
| 深挖 | 3 次以上且有决策行为 | 提炼「决策模式」和「能力与盲区」 |
| 停在初始化 | 纯传话/只说一句 | 不深挖 |

### 三层信息

- **事实** — 直接观察到的，出现一次即可写。
- **模式** — 跨事件归纳，升格为模式须同时满足三重验证。
- **边界** — 明确哪些维度无法可靠推断，写在「观察边界」段落。

#### 模式三重验证（三条同时满足才能写入）

1. **跨场景复现** — 至少 3 次独立事件（同一会议多次发言不算）
2. **生成力** — 能预测此人在新情境中的反应，不只描述过去
3. **排他性** — 描述的是这个人的特征，不是同类角色的通用行为

不满足任何一条 → 留在事件记录/待观察，不写入决策模式。

### 更新策略

**按维度组织，合并而非追加。** 新信息融入对应维度的已有描述；
不要在底部按日期追加段落；新信息推翻旧描述则直接更新，不保留旧版本。

## 二、产品画像

### 建档门槛（三条同时满足）

1. **复现性** — 在 2 条以上独立素材中出现
2. **回溯价值** — 未来写日志时需要其背景
3. **演化属性** — 有随时间变化的属性（版本、阶段、负责人）

存疑则不建档，信息留在日志条目内。

### 四维度

| 维度 | 衰减速度 |
|---|---|
| 定位与边界 — 解决什么、不解决什么、在产品矩阵里的位置 | 慢 |
| 核心能力与架构 — 技术栈、能力分层、关键指标 | 慢 |
| 战略判断与关键决策 — 谁在何时做了什么定调（标注决策人+时间） | 慢 |
| 当前态势 — 版本进度、客户管线摘要、阻塞项 | 快 |

**当前态势整体刷新，不追加。** 每次处理后重写整段并更新日期。

---

## 三、跨档案规则

- **产品决策归产品档案**：人物档案只写归纳后的决策模式，不展开产品细节。
- **唯一权威源**：职位/角色 → 人物档案；产品定位/架构 → 产品档案；会议具体讨论 → 日志条目；档案中用括号引用日志路径，不重复维护。
- **过时信息**：直接更新或删除，不在档案中追加过时警告文本。
- **产品间关系双向引用**：涉及两个以上产品的关系，两个档案都写入引用。
