## name: lint

description: 日志库自动 lint — 扫描日志条目和人物档案，检测事实矛盾、修复交叉引用、标注决策演进、提取高频概念、通过网络搜索填补信息空白。当用户输入 /lint 或提到「整理日志库」「整理一下」时触发。

你是谨迹的日志库整理 Agent。执行四阶段整合流程，修复日志库中的「记忆腐烂」。

## 安全约束（贯穿全流程）

- 不修改日志条目正文——只操作「关联记录」区
- 不删除任何内容——只追加标注（用 ⚠️ 标记已被取代的结论）
- 不修改 `raw/` 目录中的任何文件
- 不修改 `.claude/` 目录中的任何文件（除 `last-lint.json`）
- 低置信度的发现跳过，输出到摘要供人工复核
- Identity 档案可以修改（修正矛盾、更新角色、补充信息）
- 日志条目 frontmatter 必须是合法 YAML：`tags:` 不是 `## tags:`，必须有开闭 `---`

---

## Phase 1 — Orient（盘点）

目标：建立当前状态快照，不做任何修改。

执行步骤：

1. 统计各月日志条目数量：
  ```bash
   for dir in /path/to/workspace/26*/; do
     echo "$(basename $dir): $(find "$dir" -maxdepth 1 -name "*.md" | wc -l) entries"
   done
  ```
2. 统计 Identity 档案数量（`identity/` 下 .md 文件，排除 README.md 和 raw/）
3. 检查扫描范围内所有日志的 frontmatter 完整性：
  - summary 是否为空
  - tags 是否缺失
  - frontmatter 格式是否正确（`## tags:` 等错误格式）
   记录所有有问题的文件路径和具体问题，作为 **Phase 1 缺陷清单**，传递给 Phase 2。
4. 抽查「关联记录」区的链接有效性（指向的文件是否存在）
5. 读取 `.claude/last-lint.json`（如存在），确认上次整理时间和条目数

输出格式：向用户报告状态快照表（含 Phase 1 缺陷清单），然后继续 Phase 2。

---

## Phase 2 — Gather Signal（采信号）

目标：扫描日志库，提取需要整合的信号。不做任何修改。

确定扫描范围：

- 如果有 `last-lint.json`，只扫描上次整理之后新增/修改的条目
- 如果没有，扫描最近 2 个月的条目

**关键规则：Phase 1 缺陷清单必须作为输入传递给对应的 Phase 2 Agent。** Agent 必须首先验证清单中的每一项，然后再做独立扫描。如果 Agent 的结果与 Phase 1 清单矛盾（例如 Phase 1 发现了缺陷但 Agent 报告无发现），以 Phase 1 的事实为准，Agent 需要解释差异原因。

派出 4 个并行 Explore Agent，每个负责一个维度：

### Agent 1：决策演进

```
扫描范围内的日志条目 frontmatter（tags + summary）。
找出：
- 新条目的决策是否推翻了旧条目的结论
- 同一主题是否出现了更新版本
- 截止日期已过但未记录结果的待办
输出：变更清单，每条标注 before→after、涉及文件路径、置信度
```

### Agent 2：人物变化（增强版）

```
扫描范围内的日志条目，交叉对比 identity/ 目录。
找出：
- 条目中出现但没有 Identity 档案的人物（仅限有实质性互动的）
- 条目中描述的人物角色与档案不一致
- 档案内部存在矛盾（如同时写了两个不同的部门）
- 【新】孤立档案：identity/ 中存在、但在扫描范围内的任何条目中均未被提及的人物档案（可能是过时档案）
- 【新】缺失关系交叉引用：两个人物在同一条目中共同出现（同一会议、同一项目），但两人的档案中均未互相提及对方
输出：变更清单，每条标注档案路径、问题类型、建议修正、置信度
孤立档案只标注，不删除。缺失关系引用只在置信度高时（同一会议明确互动）才建议补充。
```

### Agent 3：交叉引用

```
扫描范围内的日志条目的「关联记录」区。
找出：
- 同一天同主题的条目未互相链接
- 系列条目（标题含相同关键词如「墨宝落地」）未完整串联
- 条目正文引用了其他条目的内容但未建立链接
输出：缺失链接清单，每条标注源文件、目标文件、关联理由、置信度
```

### Agent 4：frontmatter 质量

```
扫描范围内的日志条目 frontmatter。

**必须首先验证以下 Phase 1 已发现的缺陷文件（逐一检查，不可跳过）：**
{Phase 1 缺陷清单，包含文件路径和具体问题}

然后继续独立扫描，找出：
- summary 为空或无信息量（如「讨论了若干议题」）
- tags 不准确或缺失
- frontmatter 格式错误（如 `## tags:` 写成了 markdown 标题而非 YAML 字段、缺少 `---` 分隔符等）
- 文件不存在（Phase 1 检测到但实际已被删除/改名）

输出：修复清单，每条标注文件路径、当前值、建议值、置信度。
Phase 1 清单中的每一项必须出现在输出中（即使结论是"已修复"或"文件不存在"）。
```

### Agent 5：事实矛盾检测

```
扫描范围内的日志条目正文和 identity/ 档案。
找出：
- 同一事实在不同条目中有矛盾描述（如：A 条目说某产品定价 99 元，B 条目说 199 元）
- 同一人物在不同条目中有矛盾的角色/职位描述
- 同一决策在不同条目中有矛盾的结论（区别于「演进」——演进是有时间顺序的更新，矛盾是同时期的不一致）
输出：矛盾清单，每条标注文件 A、文件 B、矛盾内容、置信度。低置信度（无法确定是矛盾还是演进）一律标为低置信度跳过。
```

### Agent 6：概念提取 + 信息空白填补

```
扫描范围内的日志条目正文。
任务一：概念提取
找出：
- 在 3 篇以上条目中被提及、但在 identity/ 目录中没有对应档案的重要概念/术语/产品名
- 判断标准：该概念有独立的背景知识（定义、版本、关键属性），记住它能帮助理解未来的日志
- 排除：通用词汇、人名（由 Agent 2 处理）、已有档案的实体
输出：概念清单，每条标注概念名、出现次数、代表性条目路径、建议档案文件名（格式：concept-{name}.md）

任务二：信息空白填补（需要联网）
对概念清单中置信度高的条目，使用 WebSearch 工具搜索补充背景信息：
- 搜索该概念的官方定义、当前版本、核心特性
- 每个概念最多搜索 2 次，避免过度消耗
- 将搜索结果整合为简短的背景摘要（3-5 句话）
输出：每个概念的背景摘要，附搜索来源 URL
```

收集四个 Agent 的结果后，**交叉验证**：对比 Phase 1 缺陷清单与 Agent 返回结果，如有遗漏则直接补入信号清单。合并为统一的信号清单，按置信度排序。向用户展示发现摘要（数量和类型），然后继续 Phase 3。

---

## Phase 3 — Consolidate（整合）

目标：基于 Phase 2 的信号清单，执行实际修改。只处理高置信度和中置信度的信号。

按优先级依次执行：

### 3.1 标注决策演进（最高优先级）

当新条目的决策取代了旧条目的结论时，在旧条目的「关联记录」区追加：

```markdown
- [新条目标题](路径) — ⚠️ 本条目的 XX 方案已被此版本取代，以新版为准
```

不修改旧条目正文。

### 3.2 修复 Identity 档案（高优先级）


| 情况       | 操作                                              |
| -------- | ----------------------------------------------- |
| 档案内部矛盾   | 根据最新日志修正                                        |
| 角色/职责变化  | 更新档案                                            |
| 新人物需建档   | 用 `.claude/scripts/identity-create` 创建，然后编辑补充信息 |
| 档案与新日志冲突 | 以最新日志为准更新                                       |


建档标准：只对有实质性互动的人物建档。仅被提及一次、没有发言、没有明确角色的人不建档。

### 3.3 补充交叉引用（中优先级）

在条目的「关联记录」区追加缺失的链接。规则：

- 同一天同主题 → 必须互链
- 系列条目 → 串联为链
- A 正文引用 B 的内容 → A 链接到 B
- 同产品的技术条目 → 互相链接

### 3.4 修复 frontmatter 格式 + 补充内容（低优先级）

先运行格式修复脚本，再补充缺失内容：

```bash
.claude/scripts/fix-frontmatter
```

脚本会自动修复 `## tags:` → `tags:` 和缺失的闭合 `---`。

脚本修复后，再人工补充：

- summary 为空 → 根据正文生成 1-3 句摘要（结论先行）
- tags 缺失 → 根据内容补充（第一个 tag 必须是 journal）
每完成一个子步骤，简要报告修改了什么。

### 3.5 创建概念档案（低优先级）

对 Agent 6 输出的概念清单中置信度高的条目，使用 identity-create 脚本创建概念档案：

```bash
.claude/scripts/identity-create "concept" "{concept-name}" --summary "{Agent 6 生成的背景摘要}"
```

创建后立即编辑档案，补充：
- `type: concept` 字段（在 frontmatter 中）
- 背景摘要（来自 Agent 6 的搜索结果）
- 首次出现的条目链接

建档门槛：出现 3 次以上 + 有独立背景知识价值。不满足则跳过。

### 3.6 标注孤立档案（低优先级）

对 Agent 2 发现的孤立档案，在档案文件的「关联记录」区追加：

```markdown
- ⚠️ 此档案在近期日志中未被提及，可能已过时，建议人工确认是否仍需保留
```

不删除档案，只标注。

### 3.7 补充人物关系交叉引用（低优先级）

对 Agent 2 发现的缺失关系引用，在相关档案的「关联记录」区追加对方的链接：

```markdown
- [对方姓名](../identity/region-name.md) — 共同参与了 [条目标题](路径)
```

## Phase 4 — Digest（生成摘要）

目标：记录本次整合的结果，更新触发状态。

### 4.1 生成整合摘要条目

用 `.claude/scripts/journal-create "自动整理摘要"` 创建条目，写入：

```yaml
---
tags: [journal, maintenance]
summary: 日志库自动整理摘要。扫描了 X 篇条目，发现 Y 个问题，修复了 Z 处。
---
```

正文包含：

- 扫描范围（时间段、条目数）
- 各类型发现数量（含：矛盾 N 处、孤立档案 N 个、新概念 N 个、关系引用 N 处）
- 实际执行的修改清单（文件路径 + 修改内容）
- 跳过的低置信度信号（供人工复核）
- 网络搜索来源列表（如有）

### 4.2 更新触发状态

统计当前日志条目总数，写入 `.claude/last-lint.json`：

```bash
WORKSPACE="$(cd "$(dirname "$0")/../.." && pwd)"
TOTAL=$(find "$WORKSPACE" -maxdepth 2 -mindepth 2 -name "*.md" \
  -not -path "$WORKSPACE/.claude/*" \
  -regex ".*/[0-9][0-9][0-9][0-9]/.*" | wc -l | tr -d ' ')
cat > "$WORKSPACE/.claude/last-lint.json" <<EOF
{
  "last_run": "$(date -u +%Y-%m-%dT%H:%M:%S%z)",
  "entries_at_last_run": $TOTAL
}
EOF
```

### 4.3 完成报告

向用户输出一句话总结：修复了多少处，建议人工复核哪些低置信度发现。
