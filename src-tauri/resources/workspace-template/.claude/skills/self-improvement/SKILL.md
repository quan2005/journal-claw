# 经验积累与自检

AI 秘书的持续改进机制。把处理素材中犯的错、用户的纠正、发现的规律记录下来，下次遇到类似情况时主动应用，避免重复犯错。

## 契约

| 维度 | 声明 |
|---|---|
| **reads** | `.claude/learnings/LEARNINGS.md`、`yyMM/*.md`（自检时读最近条目） |
| **writes** | `.claude/learnings/LEARNINGS.md`、workspace `CLAUDE.md`（晋升规则时） |
| **depends** | 无 |

## 何时触发

| 场景 | 动作 |
|---|---|
| 用户纠正处理结果 | 记录学习条目（category: `correction`） |
| 用户表达偏好 | 记录学习条目（category: `preference`） |
| 处理素材失败或质量差 | 记录学习条目（category: `error`） |
| 发现可复用的规律 | 记录学习条目（category: `pattern`） |
| 用户说"查一下之前的教训" | 读取并展示相关学习记录 |
| 处理完素材后 | 运行自检协议（见 `references/self-check-protocol.md`） |
| `/lint` 运行时 | lint 自行读取 LEARNINGS.md，本 skill 不介入 |

## 核心流程

### 1. 捕获学习

用户纠正或自检发现问题时，追加条目到 `.claude/learnings/LEARNINGS.md`。

条目格式详见 `references/entry-format.md`。四种 category：

- `correction` — 用户明确纠正（"不对"、"应该是 X"、"搞错了"）
- `preference` — 用户表达偏好（"以后都这么做"、"我喜欢这样"、"别用那种格式"）
- `pattern` — 从多次处理中归纳出的规律（如"这类会议总是需要先理清参会人关系"）
- `error` — 处理失败或质量明显不达标

**捕获原则**：

- 只记录下次能用上的经验，不记录一次性的上下文
- 一条学习解决一个问题，不要把多个发现塞进一条
- 详情要写清楚"错在哪"和"对的是什么"，不要只写"出错了"
- 用户偏好要记录具体偏好内容，不要泛化（"用户喜欢简洁" → "用户要求 summary 不超过两句话"）

### 2. 回顾学习

处理新素材前，读 `.claude/learnings/LEARNINGS.md` 中 `status: active` 的条目，优先关注：

- 与当前素材类型匹配的条目（Area 字段匹配）
- priority 为 high 的条目
- 最近 30 天内记录的条目

回顾不需要向用户展示，默默应用即可。只在用户主动询问时展示学习记录。

### 3. 晋升规则

同类学习出现 **3 次**（同一 Area 下相似的 correction 或 pattern）→ 提炼为一条简洁规则，写入 workspace `CLAUDE.md` 的对应段落。

晋升流程：

1. 识别 3 条以上相关联的学习条目
2. 归纳为一条可执行的规则（一两句话）
3. 写入 workspace `CLAUDE.md`（追加到 `## 行为准则` 段落末尾）
4. 原条目标记 `status: promoted`，元数据追加 `晋升位置: CLAUDE.md`
5. 向用户简要报告晋升了什么规则

**晋升标准**：

- 规则必须是可执行的（"遇到 X 时做 Y"），不是观察（"X 经常出错"）
- 规则必须跨场景适用，不是只对某一篇素材有效
- 不确定时不晋升，继续积累

### 4. 自检协议

处理完素材后运行 5 项快速检查。详见 `references/self-check-protocol.md`。

自检不需要向用户展示过程。只在发现问题时：
- 立即修复（如 frontmatter 格式错误）
- 记录学习条目（如果是新发现的模式）

## 安全约束

- **不修改日志正文** — 自检发现的问题通过正常编辑流程修复
- **不删除学习记录** — 只追加和更新 status
- **不自动晋升** — 晋升前必须确认 3 次复现
- **LEARNINGS.md 是唯一存储** — 不在其他地方散落学习记录
- **晋升写入 CLAUDE.md 时只追加** — 不修改已有规则
