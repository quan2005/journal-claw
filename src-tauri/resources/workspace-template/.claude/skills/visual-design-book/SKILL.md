# Visual Design Book — 日志可视化说明书

## 这个 skill 做什么

读取一篇谨迹日志条目（`.md` 文件），将其内容转化为一个**单文件自包含的 HTML 页面**——一份用「逻辑推演」叙事方式呈现的可视化说明书。

**输入**：日志的 workspace 相对路径，如 `2604/12-产品评审会议.md`
**输出**：同目录下的 HTML 文件，如 `2604/12-产品评审会议-design-book.html`，并将该 HTML 路径写回日志 frontmatter 的 `sources` 字段。

## 触发方式

1. 日志详情右键菜单 →「转化为可视化设计」
2. Dock 栏输入：`/visual-design-book @2604/12-xxx.md`
3. 对话中提到"可视化设计""design book"并 @ 引用日志

## 内容类型自适应

读取日志后，先判断内容类型，再选择对应的叙事透镜：


| 类型       | 识别信号                   | 标题后缀    | Hero 称谓 | 右侧面板倾向             |
| -------- | ---------------------- | ------- | ------- | ------------------ |
| **产品设计** | 含 PRD、需求、功能、用户故事、交互、原型 | `设计说明书` | `设计说明书` | UI mockup、流程图、数据看板 |
| **会议纪要** | 含 会议、讨论、决议、待办、参会人、议题   | `会议全景`  | `会议全景`  | 决策树、责任矩阵、时间线、对比表   |
| **通用知识** | 以上均不明显                 | `知识图谱`  | `知识图谱`  | 概念关系图、分类矩阵、对比框架    |


> 判断不需要精确——看主要内容倾向即可。混合型内容取主导类型。

## 执行流程

### 步骤 1：读取日志内容

```
workspace_path + 日志相对路径 → 读取完整 markdown 内容
```

解析 frontmatter（summary、tags）和正文。这是生成说明书的素材。

### 步骤 2：判断内容类型 & 提炼叙事主线

**先判断类型**（产品设计 / 会议纪要 / 通用知识），然后用对应透镜提炼：

#### 产品设计透镜

- **北极星**：产品要解决什么核心问题？
- **推演链**：需求 → 方案 → 实现路径，概念间的因果关系
- **右侧示意**：UI mockup、交互流程、数据看板

#### 会议纪要透镜

- **北极星**：这次会议的核心议题 / 最重要的结论是什么？
- **推演链**：各议题之间的逻辑关系——不是按时间顺序平铺，而是找出议题间的因果递进（如：问题诊断 → 方案讨论 → 资源分配 → 执行计划）
- **右侧示意**：决策树、责任矩阵（谁做什么/何时）、时间线、方案对比表、风险热力图

#### 通用知识透镜

- **北极星**：这篇内容的核心洞察是什么？
- **推演链**：概念间的层级和关联
- **右侧示意**：概念关系图、分类矩阵、对比框架

### 步骤 3：规划章节结构

列出章节标题（问句形式）和每章的左右内容概要。确保：

- 章节之间有递进关系，不是并列
- 每章回答一个问题，下一章在上一章的基础上追问
- 终章能把所有线索汇聚

**会议纪要特别注意**：

- 不要按议程顺序排章节——按逻辑递进重组
- 如果会议有明确的决议和待办，终章应汇聚为「决议全景 + 行动路线图」
- 参会人、背景信息等放在 Hero 区域，不单独成章

### 步骤 4：生成 HTML

用纯 HTML + 内联 CSS 编写。参考 `references/structure-template.html` 获取完整模板。

**标题适配**：

- 产品设计：`{日志标题} · 设计说明书`
- 会议纪要：`{日志标题} · 会议全景`
- 通用知识：`{日志标题} · 知识图谱`

**输出路径**：与源日志同目录，文件名为 `{DD}-{title}-design-book.html`

### 步骤 5：关联回日志

将生成的 HTML 文件的 workspace 相对路径添加到源日志 frontmatter 的 `sources` 字段：

```yaml
---
summary: 原有摘要
tags: [原有标签]
sources:
  - 2604/12-产品评审会议-design-book.html
---
```

如果 `sources` 字段已存在，追加而非覆盖。

## 叙事结构（五层）

### 第一层：Hero（定锚）


| 元素     | 产品设计                 | 会议纪要          | 通用知识          |
| ------ | -------------------- | ------------- | ------------- |
| 标题     | `{标题} · 设计说明书`       | `{标题} · 会议全景` | `{标题} · 知识图谱` |
| Slogan | 从 summary 提炼有节奏感的一句话 | 会议核心结论，一句话    | 核心洞察，一句话      |
| 副标题    | 补充说明                 | 日期 · 参会人概要    | 补充说明          |


### 第二层：北极星愿景（战略锚点）


| 元素   | 产品设计        | 会议纪要         | 通用知识        |
| ---- | ----------- | ------------ | ----------- |
| 核心理念 | 产品要解决的核心问题  | 会议的核心议题 / 背景 | 核心概念定义      |
| 战略产出 | 产品方向 + 关键指标 | 关键决议（2-3 个）  | 核心论点（2-3 个） |
| 终极目标 | 产品愿景        | 会议期望达成的终态    | 知识应用方向      |


### 第三层：逐章推演（核心）

每章是一个「问题 → 解答」推演单元：

```
┌─────────────────────────────────────────────────┐
│  ① 编号 + 标题（问句形式）                         │
│  ② 副标题（一句话回答）                             │
│ ┌──────────────────┐  ┌──────────────────┐      │
│ │  左：推理逻辑      │  │  右：可视化呈现    │      │
│ │  • 核心概念阐述    │  │  • 产品：UI mockup│      │
│ │  • 机制说明        │  │  • 会议：决策树   │      │
│ │  • 举例论证        │  │  • 知识：关系图   │      │
│ └──────────────────┘  └──────────────────┘      │
└─────────────────────────────────────────────────┘
```

**左侧写作模式**：概念标题 → 阐释段落 → 高亮引用框 → 细节要点 → 举例框
**右侧呈现模式**（按内容类型选择）：


| 产品设计             | 会议纪要            | 通用知识  |
| ---------------- | --------------- | ----- |
| Tab栏 + 列表 = 信息展示 | 决策树 = 方案选择过程    | 概念关系图 |
| 对话气泡 = 交互过程      | 责任矩阵 = 谁做什么/何时  | 分类矩阵  |
| 节点 + 箭头 = 流程图    | 时间线 = 里程碑/排期    | 层级结构图 |
| 四象限 = 分类矩阵       | 对比表 = 方案 A vs B | 对比框架  |
| 进度条 + 数字 = 数据看板  | 风险热力图 = 风险评估    | 数据可视化 |

## 视觉设计系统

### 色彩体系

使用 CSS 变量，支持明暗双主题。默认配色是温暖文雅的金橙色系：

```
主强调色: #C8933B（金橙）
文字层级: #3D352C → #635850 → #8A8078
背景层级: #FAFAF8 → #F5F4F1 → #EFEEE9
边框: #E5E3DD
```

色系应根据日志内容调性自适应——技术类用冷色、创意类用暖色、商业类用金橙。

### 排版规范

- 字体：`Noto Sans SC` + 系统 sans-serif
- 最大宽度：1100px 居中
- 章节间距：64px
- 卡片圆角：8px

### 必备交互

1. **明暗主题切换**：右上角按钮，`data-theme` 属性切换
2. **响应式**：768px 以下左右栏变上下
3. **打印适配**：`@media print` 隐藏切换按钮、每章分页

## 暗色主题技术要点

采用**快照恢复 + 两阶段占位符替换**机制：

1. `saveSnapshot()` 保存所有 inline style 浅色原始值
2. `applyDark()` 两阶段替换：源颜色 → `@@DARK0@@` 占位符 → 深色值（防止链式碰撞）
3. `restoreSnapshot()` 直接恢复快照
4. **绝对不要用双向 colorMap**

## 叙事质量检查

- 推演链是否连贯？
- 每章标题是否是问句？副标题是否回答了问题？
- 左右对照是否匹配？
- 终章是否回扣 Hero？
- 明暗主题反复切换 3 次无色彩残留？
- HTML 文件路径已写回日志 sources？
- （会议纪要）章节是否按逻辑递进而非议程顺序？
- （会议纪要）决议和待办是否在终章汇聚？

## 输出要求

- **格式**：单个 `.html` 文件，自包含
- **语言**：跟随日志语言
- **文件名**：`{DD}-{title}-design-book.html`（与源日志同目录）
- **关联**：HTML 路径写回源日志 frontmatter `sources`
