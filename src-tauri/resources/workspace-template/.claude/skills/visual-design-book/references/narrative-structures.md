# Visual Design Book — 五层叙事结构

说明书用「逻辑推演」而不是「章节目录」组织。读者从 Hero 进入，被北极星定位，沿着逐章推演一步步建立理解，最终在终章看到所有线索汇聚。

四种内容类型**共用同一套金橙暖色设计语言**（`structure-template.html` 的 `--ac` / `Noto Sans SC` / 1100px `.page`），但在**思维结构和排版模式**上有各自的透镜。其中：

- **产品设计 / 会议纪要 / 通用知识** —— 左右对照的 `.ch-body` 网格，每章一个「问题 → 解答」推演单元
- **授课 / 分享** —— 单栏 Part 节律 + sticky NavBar + sub-label 切块，详见 `lecture-components.md`

两种排版使用完全相同的 CSS 变量、字体、最大宽度、暗色切换机制——读者打开时感受到的是**同一个设计系统**，只是信息组织方式随内容类型而变。

## 第一层：Hero（定锚）

| 元素 | 产品设计 | 会议纪要 | 通用知识 | 授课 / 分享 |
|---|---|---|---|---|
| 标题 | `{标题} · 设计说明书` | `{标题} · 会议全景` | `{标题} · 知识图谱` | `{课程 / 分享主题}` |
| Meta | slogan（一句话节奏感） | 会议核心结论 | 核心洞察 | `日期 · 机构/场合 · 讲者 · 模块数`（dot 分隔） |
| 副标题 | 补充说明 | 日期 · 参会人概要 | 补充说明 | 章节漫游（` · ` 分隔核心章节短语） |

**授课 Hero 的独特要求**：不要在主体里重复介绍讲者背景——所有讲者信息、课程元数据都压在 Hero 里，一上来就是 Part 01。

## 第二层：北极星愿景（战略锚点）

| 元素 | 产品设计 | 会议纪要 | 通用知识 | 授课 / 分享 |
|---|---|---|---|---|
| 核心理念 | 产品要解决的核心问题 | 会议的核心议题 / 背景 | 核心概念定义 | 课程的核心主张 / 理论脉络 |
| 战略产出 | 产品方向 + 关键指标 | 关键决议（2-3 个） | 核心论点（2-3 个） | 贯穿全课的底层命题（如「认知大于事实」） |
| 终极目标 | 产品愿景 | 会议期望达成的终态 | 知识应用方向 | 听众应形成的判断框架 |

授课类的北极星通常就是整门课的「一针顶破天」——找到这句就找到了整份说明书的定调。

## 第三层：逐章推演（核心）

### 产品设计 / 会议纪要 / 通用知识 —— 左右对照模式

每章是一个「问题 → 解答」推演单元：

```
┌─────────────────────────────────────────────────┐
│  ① 编号 + 标题（问句形式）                      │
│  ② 副标题（一句话回答）                        │
│ ┌──────────────────┐  ┌──────────────────┐      │
│ │  左：推理逻辑     │  │  右：可视化呈现    │      │
│ │  • 核心概念阐述   │  │  • 产品：UI mockup │      │
│ │  • 机制说明       │  │  • 会议：决策树    │      │
│ │  • 举例论证       │  │  • 知识：关系图    │      │
│ └──────────────────┘  └──────────────────┘      │
└─────────────────────────────────────────────────┘
```

**左侧写作模式**：概念标题 → 阐释段落 → 高亮引用框 → 细节要点 → 举例框

**右侧呈现模式（按内容类型选择）**：

| 产品设计 | 会议纪要 | 通用知识 |
|---|---|---|
| Tab 栏 + 列表 = 信息展示 | 决策树 = 方案选择过程 | 概念关系图 |
| 对话气泡 = 交互过程 | 责任矩阵 = 谁做什么/何时 | 分类矩阵 |
| 节点 + 箭头 = 流程图 | 时间线 = 里程碑/排期 | 层级结构图 |
| 四象限 = 分类矩阵 | 对比表 = 方案 A vs B | 对比框架 |
| 进度条 + 数字 = 数据看板 | 风险热力图 = 风险评估 | 数据可视化 |

### 授课 / 分享 —— 单栏节律模式

授课的每章不走左右对照。因为讲者的叙述本身就是左右耦合的（理论 + 案例 + 引用），硬拆会失去课堂节奏。采用：

```
┌─────────────────────────────────────────────────┐
│ Part 0X     {本节主标题}                        │
│ ────────────────────────────────────────────    │
│ summary-line（立意，金色左边线）                 │
│                                                 │
│ sub-label「演进全景」                            │
│   [SVG 演进图 或 info-grid N要素]                │
│                                                 │
│ sub-label「各阶段核心命题」                      │
│   [data-table 矩阵展开]                          │
│                                                 │
│ sub-label「案例」                                │
│   [case-card]                                   │
│                                                 │
│ quote 收束（讲者原话）或 diff-alert 提示          │
└─────────────────────────────────────────────────┘
```

**组件选择指南**（详细语义和 HTML 模板见 `lecture-components.md`）：

| 当你要表达 | 用这个组件 |
|---|---|
| 理论演进（A → B → C → ...） | SVG 节点链，最后一个节点高亮 |
| N 要素 / N 特征 / N 维度 | `.info-grid.col3~5` + `.icard` |
| 行×列都有信息（对比多个维度） | `.data-table`，末行可高亮 |
| 二元对立 / A vs B | `.compare` + 两个 `.ccard`（异色边线） |
| 层次递进 / 博弈回合 / 多步骤 | `.timeline` + 不同色的 `tl-dot` |
| 讲者原话 / 经典结论 | `.logic-quote`（金橙 `--quote-bg`） |
| 整理者提醒 / 易错点 | `.diff-alert`（⚠ / 💡 / ✓ 图标） |
| 案例详细剖析 | `.case-card`（CASE STUDY 标签） |
| 2×2 战略矩阵（白/灰/黑地带等） | `.info-grid.col2` + 异色背景 icards |

## 第四层：度量 / 治理章节组（可选）

产品设计类日志可能包含「如何度量成功 / 如何治理风险」的章节组。会议纪要如果涉及 KPI / 目标跟踪也可用。

**授课不走这一层**——授课的「治理」自然地融进案例与 diff-alert，不单独成章。

## 第五层：终章收束

| 元素 | 产品设计 | 会议纪要 | 通用知识 | 授课 / 分享 |
|---|---|---|---|---|
| 汇聚方式 | 所有推演路径回扣 Hero | 决议全景 + 行动路线图 | 知识脉络总览 | 全局语录矩阵 + 能力 / 框架合成图 |
| 右侧 | 产品全景 mockup | 甘特图 / 里程碑时间线 | 概念全景图 | 全局 quote 方阵（讲者金句汇编）|

授课终章应做两件事：
1. 把全课最重要的 **3-6 句讲者原话**合并成一个大 `.quote` 方阵（`font-size:0.95em;line-height:2;`）
2. 给出一个**能力 / 框架合成图**（可选形态：首字母缩略词方阵、四象限矩阵、N 维雷达图、分层能力模型等），让听众离开时带走一个能复用的判断工具

## 章节规划原则

- 章节之间有**递进关系**，不是并列
- 每章回答一个问题（或推进一个理论），下一章在上一章的基础上追问
- 终章能把所有线索汇聚
- **会议纪要不要按议程顺序排章节** — 按逻辑递进重组，议程是会议的顺序，逻辑是说明书的顺序
- **授课可以基本遵循讲者的授课顺序**——优秀的讲者本身已经按逻辑推演讲课，不需要重排；但要**合并讲者的数字编号与你的 Part 编号**，用 `Part 01` `Part 02` 统一编号，不用讲者说的「这是第三点」做标题
- 参会人 / 讲者背景 / 课程元数据等放在 Hero 区域，不单独成章
